PII Extraction Engine for refund tickets: Technical Documentation
Overview
The Problem: Extracting Personally Identifiable Information (PII)—specifically Name, Email, Refund Amount, and Currency—from unstructured customer support tickets.
The Solution: An automated data extraction API utilizing Small Language Models (SLMs) to output structured JSON.
Why SLMs?
Data Privacy: Tickets contain sensitive PII. Local processing ensures data never leaves the internal network.
Latency: Eliminates network round trips associated with cloud API calls.
Cost per ticket: Zero cost per ticket compared to cloud-based token billing.
Hardware Configuration
Processor: Intel Core i5
Memory: 16GB RAM (Integrated)
Deployment: Local/Offline via Ollama
Phase 1: Inference Metrics
The Problem: Automation pipelines require high throughput. A model that is too slow creates a system bottleneck.
The Test: Evaluated Llama 3.2 (1B), Qwen 2.5 (1.5B), Gemma 3 (1B), and DeepSeek-R1 for Time to First Token (TTFT) and Tokens Per Second (TPS).
The Result: * DeepSeek-R1 was disqualified. Chain-of-Thought reasoning induced unacceptable latency.
The remaining three models achieved speed parity, stabilizing between 18.5 and 22.3 TPS.
All models experienced a cold start penalty. The first prompt required 3.0+ seconds. By the second prompt, TTFT decreased by 70% as the model remained loaded in shared memory.
Phase 2: Structural Determinism
The Problem: The pipeline requires perfectly structured JSON data. Creative text generation breaks database schemas.
The Test: Model temperature was set to 0 to enforce structure. Models were tested against a 24-ticket dataset containing severe edge cases, including:
Missing fields (name, email, amount)
Multiple amounts or emails in one text
Forwarded chains and quoted replies
Agent signature noise
Negation (denied refunds)
Multilingual text and OCR noise
Amounts written in words
Currency symbols versus ISO codes
Extra JSON bait and markdown noise
The Result: * Llama 3.2 (1B) was rejected for data omission.
Gemma 3 (1B) was rejected for data hallucination.
Qwen 2.5 (1.5B) was selected as the primary model for demonstrating the highest structural baseline.
Phase 3: Prompt Engineering
The Problem: Qwen 2.5 initially scored 58.3% accuracy. It failed to map currency symbols (like $) to text and failed complex formatting.
The Solution: The system prompt was updated to explicitly map currency symbols to 3-letter ISO codes. Few-shot prompting was implemented, providing three distinct input/output JSON examples directly in the instructions.
The Result: Accuracy increased to 79.1%.
Phase 4: The Retry & Cascade Architecture
The Problem: The 1.5B parameter model reached a hard accuracy limit at 79.1%. It occasionally outputs invalid JSON syntax. It also struggles with complex logic (like quoted email threads), causing it to miss core data.
The Solution: A two-step safety net built directly into the API routing logic.
The Retry Mechanism: When the primary model (Qwen 1.5B) outputs broken JSON, the system catches the error. It feeds the exact error text back to the model and reprompts it to fix the syntax.
The LLM Cascade (Mitigation): If the retry fails, or if the model outputs valid JSON but misses core business data (like Name or Amount), the system intercepts the ticket. It automatically mitigates the query by routing it to a heavier fallback model: Llama 3.2 (3B).
The Result: The 3B model successfully resolved the complex edge cases. This architecture pushed the final API accuracy to 83.3%. It processes most tickets at high speed and only uses heavier compute when absolutely necessary.
